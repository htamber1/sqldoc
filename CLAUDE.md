# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`sqldoc` is a CLI that connects to a **SQL Server** database, extracts schema metadata, uses an LLM to generate plain-English descriptions of each table and column, and renders a single self-contained HTML documentation file. It has grown into an **eight-command** platform. The CLI is a command group: **`sqldoc doc`** (documentation), **`sqldoc scan`** (PII/compliance scan), **`sqldoc health`** (DMV performance/health + unused-objects detector), **`sqldoc quality`** (aggregate data profiling), **`sqldoc intel`** (schema intelligence), **`sqldoc insights`** (AI-powered: NL-to-SQL, anomalies, glossary, relationship inference), **`sqldoc comply`** (HIPAA/GDPR/PCI-DSS reports, data lineage, per-principal access audit), and **`sqldoc agent`** (persistent background monitoring daemon); a `DefaultGroup` routes `sqldoc <options>` (no subcommand) to `doc` for backward compatibility. Entry point is `sqldoc.cli:cli`.

### sqldoc agent (v1.7.0) — persistent background monitoring
New `sqldoc/agent/` package + `sqldoc agent` command group. Turns sqldoc into a living monitoring system.
- **`store.py` `AgentStore`** — thread-safe SQLite (`~/.sqldoc/agent.db`, WAL, connection-per-call): per-db snapshots, AI caches, rendered docs, run history, event timeline, metrics time-series, PII snapshots.
- **`config.py`** — `parse_agent_config()` reads the `.sqldoc.yml` `agent:` section (interval, dashboard_port, mode/model, per-db connection resolution via each dialect's `build_connection_string`, notifications). `agent` added to `CONFIG_KEYS`.
- **`notify.py`** — `Notifier` → Slack webhook (`requests`) + email (`smtplib`), `on` allowlist, per-channel isolation. Transports are module-level for mocking.
- **`poller.py`** — `poll_database()` = the per-db pass: adapter extract → snapshot diff → AI enrich (cache reuse ⇒ only changed objects call the LLM) → render+store HTML → PII scan + 0-100 `pii_score()` + drift → health (capability-gated) + degradation → metric + timeline events + notifications. Wrapped so a failure is a recorded failed run, never a crash.
- **`dashboard.py`** — stdlib `http.server` (no dep). Pure render fns (`render_overview`/`render_db_page`) + handler + `make_server`. Routes: `/`, `/db/<name>`, `/db/<name>/doc`, `/api/overview`. Dark theme, inline SVG sparklines.
- **`daemon.py`** — `run_daemon()` = dashboard thread + one `poller_loop` thread per db, `stop_event.wait(interval)` for responsive shutdown; `watch_stop_flag()` helper.
- **`cli.py`** — `agent start` (spawns detached `agent _run` daemon; PID/log in `~/.sqldoc/`; `--foreground`), `stop` (graceful stop-flag + terminate fallback), `status`, `logs [-n][-f]`, hidden `_run` (SIGTERM/SIGINT + stop-flag watcher). **Windows-safe `pid_alive`** (never `os.kill(pid,0)` — that TERMINATES on Windows; uses `OpenProcess`/`GetExitCodeProcess`).
- **Tests (47):** store, config+notify, poller (schema-change/new-PII/health-degradation/error), dashboard+daemon (live ephemeral-port server, lifecycle), CLI incl. a **real end-to-end run** against a temp SQLite DB. **Live-smoke-tested the real spawned daemon** (start→status→dashboard→logs→stop) against a Chinook-style SQLite DB. Packaged `sqldoc.agent` in `pyproject`.
- **Deferred for the agent:** auth on the dashboard (it's localhost-only + unauthenticated by design); PDF/markdown in the dashboard; webhook retry/queue; per-db enable/disable at runtime; a systemd/Windows-service wrapper. Not blockers.

### Multi-database (v1.5.x, shipped + live-validated)
**Delivered:** **PostgreSQL, MySQL, and Azure SQL** alongside SQL Server, through a common `DatabaseAdapter` interface. `doc`/`scan`/`intel`/`insights` run on all four dialects; `health`/`quality`/`comply` access-audit stay SQL-Server/Azure-only (their DMV/aggregate/`sys.database_permissions` SQL isn't ported — the commands refuse other dialects via a `Capabilities` guard with a clear message). Shipped as v1.5.0; **v1.5.1 live-validated** the Postgres + MySQL adapters against real **Pagila** (PostgreSQL 16) and **Sakila** (MySQL 8) in Docker.

**Live-validation (v1.5.1) — two real bugs found + fixed:**
- **Postgres partitioned tables** — Pagila's `payment` is a declarative-partition parent; the table query filtered `relkind='r'` only, so it surfaced the 7 physical partitions and dropped the logical parent. Fixed: include `relkind IN ('r','p')`, exclude `relispartition`, clamp the parent's `-1` reltuples to 0. (15 tables, not 21.)
- **MySQL cursor** — `mysql-connector-python` 9.x removed the `named_tuple` cursor (raised `unexpected keyword argument 'named_tuple'`). Switched `MySQLAdapter` to a `dictionary=True` cursor with `row["col"]` access; `conftest.FakeRow.__getitem__` now also accepts string keys so one fake mimics both pyodbc rows and dict cursors.
- Validated end-to-end: Pagila → 15 tables/7 views/9 funcs (PK/FK with RESTRICT/CASCADE, grouped indexes, `last_updated` triggers, real row counts); Sakila → 16 tables/7 views/6 procs (PK/FK, ENUM/SET/YEAR types, ins/upd/del triggers, composite UNIQUEs, OUT params). `scan` + `intel` also run clean on both; `health` correctly refused on both via the capability guard. **Repro:** `postgres:16` + Pagila, `mysql:8` + Sakila (jOOQ mirror), drivers `psycopg2-binary` + `mysql-connector-python`.

## Running

Packaged with `pyproject.toml` (setuptools); `pip install .` (or `-e .`) installs the **`sqldoc`** console command (entry point `sqldoc.cli:main`). You can still run it as a module from the repo root using the checked-in venv:

```bash
venv/Scripts/python.exe -m sqldoc.cli --server <host> --database <db> --username <user> --password <pw> --output docs.html
```

Key options (see `sqldoc/cli.py`):
- `--mode local|cloud` — `local` (default) calls Ollama at `http://localhost:11434`; `cloud` calls the Anthropic API.
- `--model` — model name. **Default is `None` and resolved per-mode** in `main()`: `llama3.1:8b` for local, `claude-haiku-4-5` for cloud. This split exists so the local Ollama tag never leaks into a cloud API call. Explicitly passing `--model` overrides for whichever backend is active; it is threaded all the way through to `_call_anthropic(prompt, model)` / `_call_ollama(prompt, model)`.
- `--no-ai` — skip all LLM calls, emit schema-only docs. Use this to iterate on extraction/rendering without a running LLM (also the fastest smoke test of the CLI plumbing).
- `--schemas` — comma-separated schema allowlist; filtering happens in `cli.py` *after* full extraction, and applies to tables, views, and procedures alike.
- `--concurrency` — parallel AI calls during enrichment (1-64, default 8). Threaded to all three `enrich_*` functions, which run their per-object calls through a `ThreadPoolExecutor`.
- `--connection-string` — full ODBC connection string as an alternative to `--server/--database/--username/--password`; takes precedence over them. The database name is parsed out (`DATABASE=`/`Initial Catalog=`) for labeling + snapshot/cache filenames. The extractor now takes a single connection string (`build_connection_string()` assembles it from parts).
- `--cache` / `--no-cache` — AI description cache (default `.sqldoc-cache/<database>.json`). Reuses descriptions for objects whose structural signature is unchanged; gitignored.
- `--config` — path to a `.sqldoc.yml` (default `.sqldoc.yml` in cwd if present). Any option can be set there; an explicit CLI flag overrides the config, which overrides the built-in default. Connection flags are optional when supplied via config.
- `--yes` / `-y` — bypass the cloud confirmation prompt (below) for non-interactive/CI runs.

**Privacy posture — local-first by design.** Local mode is the default; nothing leaves the network unless `--mode cloud` is explicitly chosen. The banner prints a `Privacy:` line stating the egress posture every run. `--mode cloud` additionally prints a warning and blocks on an interactive `click.confirm` (defaults to "no") before any network call — `--yes` skips only the prompt, not the warning. Note the data boundary: **only schema metadata** (table/column names, types, keys, row counts, existing `MS_Description` text) is ever sent to Anthropic — the extractor queries only `sys.*` catalog views and never reads table row data.

`--mode cloud` needs `ANTHROPIC_API_KEY`, loaded from `.env` via `python-dotenv` (`load_dotenv()` runs at import in `cli.py`).

## Architecture

A linear pipeline, one module per stage, orchestrated by `cli.py`:

1. **`adapters/` + `extractor.py`** — the metadata-extraction layer. **`adapters/base.py`** defines the shared-currency dataclasses (`Table`, `Column`, `Index` attached to a `Table`, `View`, `Parameter`, `StoredProcedure`), a `Capabilities` advertisement, and the `DatabaseAdapter` ABC. **`adapters/sqlserver.py` `SqlServerAdapter`** is the DB-facing code today: `extract_metadata()` queries `sys.*` catalog views (tables + row counts, a per-table `sys.columns` join for PK/FK/`extended_properties`=`MS_Description`, and a per-table `sys.indexes`/`sys.index_columns` query grouping key vs included columns); `extract_views()`/`extract_procedures()` add views (with `sys.sql_modules.definition`) and procedures (with `sys.parameters`). Any real `MS_Description` is captured here — the AI stage only fills descriptions left empty. **`extractor.py`** is a thin back-compat shim re-exporting the dataclasses + `build_connection_string`/`get_connection`/`extract_*` (delegating to `SqlServerAdapter`); `adapters/__init__.py` `get_adapter(cs, dialect)` picks the adapter by `--dialect`/auto-detection. NOTE: `health`/`quality`/`comply`/`pii` still run their own dialect-specific SQL via `extractor.get_connection` (see the "deferred" note in the v1.5.0 plan).

2. **`ai.py`** — `enrich_tables()`/`enrich_views()`/`enrich_procedures()` mutate the objects in place, setting `.description`. Two backends behind a `mode` switch: `_call_ollama` (HTTP to localhost) and `_call_anthropic` (SDK, shared lazily-created client, default model `claude-haiku-4-5`). Each `enrich_*` builds a flat list of independent per-object work units and runs them through a `ThreadPoolExecutor` (`--concurrency`, default 8) — each unit writes only its own object's `.description`, so there is no cross-thread shared state. View/proc prompts are **metadata-only** (names + columns/params, never the SQL definition) to keep the cloud data boundary unchanged.

3. **Renderers (one per format, same signature `(database, tables, output, views, procedures)`)** — `cli.py` picks one from `--format`/extension:
   - **`renderer.py`** `render_html()` — groups each object type by schema and renders the single `HTML_TEMPLATE` via an **autoescaping** Jinja `Environment` (all CSS inlined) to one standalone file. Autoescaping matters: view/proc definitions contain `<`/`>`/`&`.
   - **`markdown_renderer.py`** `render_markdown()` — single `.md` for GitHub wikis (TOC + anchor links, escaped table cells, fenced SQL).
   - **`pdf_renderer.py`** `render_pdf()` — `fpdf2` multi-page PDF; imported lazily so the other formats don't require it; Latin-1 text sanitization.

4. **`snapshot.py`** — orthogonal to rendering. `build_snapshot()` serializes the schema **structure** (not descriptions/rows) to JSON; `diff_snapshots()` + `iter_diff_lines()` produce the git-diff-style change report that `cli.py` prints (colored) and then re-saves the snapshot. Runs after schema-filtering, before enrichment, so it works with `--no-ai`.

5. **`pii.py` + `pii_renderer.py`** — the `sqldoc scan` path (independent of the doc pipeline; reuses `extract_metadata`). `pii.py` holds the `PII_CATEGORIES` catalog (name patterns → severity + regulations + action), a camelCase-aware matcher (`\b` patterns match tokenized names, others match the compact name), `scan_tables()` → `Finding`s with risk/confidence, and `confirm_with_sampling()` (opt-in `--sample`: reads ≤5 values per column, AI-confirms, **never stores samples**). `pii_renderer.render_pii_html()` writes the compliance report (dashboard, risk filter, CSV export via `tojson`). If you add a PII category, edit only `PII_CATEGORIES`.

The dataclasses flow through unchanged: extractor builds them → ai enriches them → renderers read them. If you add a **field**, touch the extractor, `ai.py` (if it needs a description), **all three renderers**, and `snapshot.py` (if it's a structural field worth diffing). If you add a new **object type**, also thread it through `cli.py` (extract → schema-filter → snapshot → enrich → render).
