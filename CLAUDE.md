# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`sqldoc` is a **SQL Server infrastructure platform** (as of v2.0.0): it began as a database documentation CLI and now covers instance-level and infrastructure monitoring too. It connects to a database, extracts schema metadata, uses an LLM to generate plain-English descriptions, and renders self-contained HTML ŌĆö and it also monitors the whole SQL Server instance. It has grown into an **eighteen-command** platform (plus a VS Code extension). The CLI is a command group: **`sqldoc doc`** (documentation), **`sqldoc scan`** (PII/compliance scan), **`sqldoc health`** (DMV performance/health + unused-objects detector), **`sqldoc quality`** (aggregate data profiling), **`sqldoc intel`** (schema intelligence + linked-server network mapping), **`sqldoc insights`** (AI-powered: NL-to-SQL, anomalies, glossary, relationship inference), **`sqldoc comply`** (HIPAA/GDPR/PCI-DSS reports, data lineage, per-principal access audit, cross-database board report), **`sqldoc dbt`** (unify dbt model metadata with the live schema), **`sqldoc server`** (instance-level health + SQL Agent jobs + backups + tempdb), **`sqldoc logs`** (ERRORLOG reader), **`sqldoc secure`** (security vulnerability scanner + score), **`sqldoc waits`** (wait-statistics analysis + AI), **`sqldoc ha`** (HA/replication monitoring), **`sqldoc deadlocks`** (deadlock analysis + SVG + AI), **`sqldoc plans`** (query execution-plan analyzer + AI), **`sqldoc capacity`** (capacity planning from agent history), **`sqldoc baseline`** (performance baseline + anomaly detection), and **`sqldoc agent`** (persistent background monitoring daemon, now incl. server/backup/HA/tempdb + AI natural-language alerts); a `DefaultGroup` routes `sqldoc <options>` (no subcommand) to `doc` for backward compatibility. Entry point is `sqldoc.cli:cli`.

## Running

Packaged with `pyproject.toml` (setuptools); `pip install .` (or `-e .`) installs the **`sqldoc`** console command (entry point `sqldoc.cli:main`). You can still run it as a module from the repo root using the checked-in venv:

```bash
venv/Scripts/python.exe -m sqldoc.cli --server <host> --database <db> --username <user> --password <pw> --output docs.html
```

Key options (see `sqldoc/cli.py`):
- `--mode local|cloud` ŌĆö `local` (default) calls Ollama at `http://localhost:11434`; `cloud` calls the Anthropic API.
- `--model` ŌĆö model name. **Default is `None` and resolved per-mode** in `main()`: `llama3.1:8b` for local, `claude-haiku-4-5` for cloud. This split exists so the local Ollama tag never leaks into a cloud API call. Explicitly passing `--model` overrides for whichever backend is active; it is threaded all the way through to `_call_anthropic(prompt, model)` / `_call_ollama(prompt, model)`.
- `--no-ai` ŌĆö skip all LLM calls, emit schema-only docs. Use this to iterate on extraction/rendering without a running LLM (also the fastest smoke test of the CLI plumbing).
- `--schemas` ŌĆö comma-separated schema allowlist; filtering happens in `cli.py` *after* full extraction, and applies to tables, views, and procedures alike.
- `--concurrency` ŌĆö parallel AI calls during enrichment (1-64, default 8). Threaded to all three `enrich_*` functions, which run their per-object calls through a `ThreadPoolExecutor`.
- `--connection-string` ŌĆö full ODBC connection string as an alternative to `--server/--database/--username/--password`; takes precedence over them. The database name is parsed out (`DATABASE=`/`Initial Catalog=`) for labeling + snapshot/cache filenames. The extractor now takes a single connection string (`build_connection_string()` assembles it from parts).
- `--cache` / `--no-cache` ŌĆö AI description cache (default `.sqldoc-cache/<database>.json`). Reuses descriptions for objects whose structural signature is unchanged; gitignored.
- `--config` ŌĆö path to a `.sqldoc.yml` (default `.sqldoc.yml` in cwd if present). Any option can be set there; an explicit CLI flag overrides the config, which overrides the built-in default. Connection flags are optional when supplied via config.
- `--yes` / `-y` ŌĆö bypass the cloud confirmation prompt (below) for non-interactive/CI runs.

**Privacy posture ŌĆö local-first by design.** Local mode is the default; nothing leaves the network unless `--mode cloud` is explicitly chosen. The banner prints a `Privacy:` line stating the egress posture every run. `--mode cloud` additionally prints a warning and blocks on an interactive `click.confirm` (defaults to "no") before any network call ŌĆö `--yes` skips only the prompt, not the warning. Note the data boundary: **only schema metadata** (table/column names, types, keys, row counts, existing `MS_Description` text) is ever sent to Anthropic ŌĆö the extractor queries only `sys.*` catalog views and never reads table row data.

`--mode cloud` needs `ANTHROPIC_API_KEY`, loaded from `.env` via `python-dotenv` (`load_dotenv()` runs at import in `cli.py`).

## Dependencies

Pinned in `requirements.txt`: `click`, `pyodbc`, `anthropic`, `jinja2`, `requests`, `python-dotenv`, `PyYAML` (also installed in the checked-in `venv/`). ## Architecture

A linear pipeline, one module per stage, orchestrated by `cli.py`:

1. **`adapters/` + `extractor.py`** ŌĆö the metadata-extraction layer. **`adapters/base.py`** defines the shared-currency dataclasses (`Table`, `Column`, `Index` attached to a `Table`, `View`, `Parameter`, `StoredProcedure`), a `Capabilities` advertisement, and the `DatabaseAdapter` ABC. **`adapters/sqlserver.py` `SqlServerAdapter`** is the DB-facing code today: `extract_metadata()` queries `sys.*` catalog views (tables + row counts, a per-table `sys.columns` join for PK/FK/`extended_properties`=`MS_Description`, and a per-table `sys.indexes`/`sys.index_columns` query grouping key vs included columns); `extract_views()`/`extract_procedures()` add views (with `sys.sql_modules.definition`) and procedures (with `sys.parameters`). Any real `MS_Description` is captured here ŌĆö the AI stage only fills descriptions left empty. **`extractor.py`** is a thin back-compat shim re-exporting the dataclasses + `build_connection_string`/`get_connection`/`extract_*` (delegating to `SqlServerAdapter`); `adapters/__init__.py` `get_adapter(cs, dialect)` picks the adapter by `--dialect`/auto-detection. NOTE: `health`/`quality`/`comply`/`pii` still run their own dialect-specific SQL via `extractor.get_connection` (see the "deferred" note in the v1.5.0 plan).

2. **`ai.py`** ŌĆö `enrich_tables()`/`enrich_views()`/`enrich_procedures()` mutate the objects in place, setting `.description`. Two backends behind a `mode` switch: `_call_ollama` (HTTP to localhost) and `_call_anthropic` (SDK, shared lazily-created client, default model `claude-haiku-4-5`). Each `enrich_*` builds a flat list of independent per-object work units and runs them through a `ThreadPoolExecutor` (`--concurrency`, default 8) ŌĆö each unit writes only its own object's `.description`, so there is no cross-thread shared state. View/proc prompts are **metadata-only** (names + columns/params, never the SQL definition) to keep the cloud data boundary unchanged.

3. **Renderers (one per format, same signature `(database, tables, output, views, procedures)`)** ŌĆö `cli.py` picks one from `--format`/extension:
   - **`renderer.py`** `render_html()` ŌĆö groups each object type by schema and renders the single `HTML_TEMPLATE` via an **autoescaping** Jinja `Environment` (all CSS inlined) to one standalone file. Autoescaping matters: view/proc definitions contain `<`/`>`/`&`.
   - **`markdown_renderer.py`** `render_markdown()` ŌĆö single `.md` for GitHub wikis (TOC + anchor links, escaped table cells, fenced SQL).
   - **`pdf_renderer.py`** `render_pdf()` ŌĆö `fpdf2` multi-page PDF; imported lazily so the other formats don't require it; Latin-1 text sanitization.

4. **`snapshot.py`** ŌĆö orthogonal to rendering. `build_snapshot()` serializes the schema **structure** (not descriptions/rows) to JSON; `diff_snapshots()` + `iter_diff_lines()` produce the git-diff-style change report that `cli.py` prints (colored) and then re-saves the snapshot. Runs after schema-filtering, before enrichment, so it works with `--no-ai`.

5. **`pii.py` + `pii_renderer.py`** ŌĆö the `sqldoc scan` path (independent of the doc pipeline; reuses `extract_metadata`). `pii.py` holds the `PII_CATEGORIES` catalog (name patterns ŌåÆ severity + regulations + action), a camelCase-aware matcher (`\b` patterns match tokenized names, others match the compact name), `scan_tables()` ŌåÆ `Finding`s with risk/confidence, and `confirm_with_sampling()` (opt-in `--sample`: reads Ōēż5 values per column, AI-confirms, **never stores samples**). `pii_renderer.render_pii_html()` writes the compliance report (dashboard, risk filter, CSV export via `tojson`). If you add a PII category, edit only `PII_CATEGORIES`.

The dataclasses flow through unchanged: extractor builds them ŌåÆ ai enriches them ŌåÆ renderers read them. If you add a **field**, touch the extractor, `ai.py` (if it needs a description), **all three renderers**, and `snapshot.py` (if it's a structural field worth diffing). If you add a new **object type**, also thread it through `cli.py` (extract ŌåÆ schema-filter ŌåÆ snapshot ŌåÆ enrich ŌåÆ render).
